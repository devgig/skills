# Example — UserService (reverse-engineered)

This directory is a **fully-worked spec** that, fed back through
`scaffold-csharp-microservice`, produces a service functionally equivalent
to the existing UserService reference at `~/repos/UserService`.

It's laid out in the [GitHub Spec Kit](https://github.com/github/spec-kit)
style — each artifact has one job:

| File | What it defines | Maps to in the generated code |
|---|---|---|
| [`spec.md`](./spec.md) | Intent: identity, user stories, scope, business rules, success criteria | Solution name, `catalog-info.yaml`, handler invariants |
| [`plan.md`](./plan.md) | Technical plan: architecture, tech choices, file layout, deployment | `Program.cs` wiring, NuGet packages, Dockerfile |
| [`data-model.md`](./data-model.md) | Entities, fields, indexes, query methods | `src/Domain/Entities/`, `src/Domain/Repositories/` |
| [`contracts/users-api.yaml`](./contracts/users-api.yaml) | HTTP contract for the User resource | `src/Service/Endpoints/`, `src/Service/Models/`, `src/Service/Validators/`, `src/Application/{Commands,Queries,Handlers}/` |
| [`tasks.md`](./tasks.md) | Ordered agent task list | What the skill ticks off as it runs |
| [`tests/`](./tests/) | Acceptance tests | `tests/UnitTests/`, `tests/IntegrationTests/` |
| [`catalog-info.yaml`](./catalog-info.yaml) | Backstage descriptor (output) | Generated by the skill from `spec.md` §Service |

Open these side-by-side with the real `~/repos/UserService` to see how each
spec section projects into code.

## Deliberate gaps vs. the real UserService

The real repo has extras this example deliberately omits (so the example
stays focused on the core pattern):

- **Kafka consumer project** — the real repo ships a `Consumer/` project
  and `ConsumerIntegrationTests/`; those are out of scope for this skill.
- **`Application.Services.UserService`** wrapper — the real repo wraps
  MediatR behind a service class; this example talks to `ISender` directly.
- **`Mappers/`** — the real repo has explicit DTO mappers; this example
  uses record constructors inline.

Adopt these additions later when your service actually needs them. Don't
import them preemptively.
